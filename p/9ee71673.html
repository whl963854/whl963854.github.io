    Linux线程同步对象

    Linux互斥体

    Linux互斥体与Windows的临界区对象用法很相似，一般也是通过限制多个线程同时执行某段代码来保护资源的。在Linux中，互斥体对象（mutex）用于保护共享资源，以防止多个线程同时访问或修改该资源。Linux提供了pthread_mutex_t数据结构来表示互斥体对象，并提供了一些函数来进行初始化和操作互斥体。

    有两种方式可以初始化互斥体对象：

    1. 使用PTHREAD_MUTEX_INITIALIZER宏进行初始化：
    pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

    这种方式适用于静态分配的互斥体对象，直接通过宏进行初始化。

    1. 使用pthread_mutex_init函数进行初始化：
    pthread_mutex_t mutex;
pthread_mutex_init(&mutex, NULL);

    这种方式适用于动态分配的互斥体对象或需要设置额外属性的情况。pthread_mutex_init函数用于初始化互斥体，第一个参数是指向互斥体对象的指针，第二个参数是指向pthread_mutexattr_t结构的指针，用于设置互斥体的属性。如果不需要设置属性，可以将第二个参数设为NULL。
    如果函数执行成功，则会返回 0；如果执行失败，则会返回 1 个错误码。
    需要注意的是，在使用互斥体保护资源时，需要使用pthread_mutex_lock函数进行加锁，保证只有一个线程可以进入临界区执行代码。在临界区执行完后，使用pthread_mutex_unlock函数进行解锁，允许其他线程进入临界区。

    当我们不再需要一个互斥体对象时，可以使用pthread_mutex_destroy函数将其销毁。该函数的函数签名如下：

    int pthread_mutex_destroy(pthread_mutex_t *mutex);

    参数mutex是指向要销毁的互斥体对象的指针。调用pthread_mutex_destroy函数后，互斥体对象将不再可用。
    如果函数执行成功，则会返回0；如果执行失败，则会返回一个错误码表明出错的原因。这里需要注意两点。

    • 无须销毁使用PTHREAD_MUTEX_INITIALIZER初始化的互斥体。
    • 不要销毁一个已经加锁或正在被条件变量使用的互斥体对象，当互斥体处于已加锁状态或者正在和条件变量配合使用时，调用pthread_mutex_destroy函数会返回EBUSY错误。

    以下代码演示了尝试销毁一个被锁定的互斥体对象的过程：

    #include <pthread.h>
#include <stdio.h>

pthread_mutex_t mutex;

void* threadFunc(void* arg) {
    // 加锁互斥体
    pthread_mutex_lock(&mutex);
    printf("Thread locked mutex\n");
    
    // 模拟一段耗时操作
    sleep(2);
    
    // 解锁互斥体
    pthread_mutex_unlock(&mutex);
    printf("Thread unlocked mutex\n");
    
    return NULL;
}

int main() {
    // 初始化互斥体
    pthread_mutex_init(&mutex, NULL);
    
    // 创建线程
    pthread_t thread;
    pthread_create(&thread, NULL, threadFunc, NULL);
    
    // 等待一段时间，让线程获得锁
    sleep(1);
    
    // 销毁互斥体（尝试）
    int result = pthread_mutex_destroy(&mutex);
    if (result == 0) {
        printf("Mutex destroyed successfully\n");
    } else {
        printf("Failed to destroy mutex\n");
    }
    
    // 等待线程结束
    pthread_join(thread, NULL);
    
    return 0;
}

    在上面的代码中，我们创建了一个线程，并在其中锁定了互斥体。然后，在主线程中尝试销毁该互斥体对象。运行代码时，会发现尝试销毁失败，因为互斥体被锁定，无法销毁。
    这是因为互斥体对象在被锁定状态下是不允许被销毁的。如果尝试销毁一个被锁定的互斥体，pthread_mutex_destroy函数将返回一个非零的错误码，表示销毁失败。
    在实际应用中，应该确保在销毁互斥体之前，先释放互斥体的所有锁定。只有当互斥体处于未锁定状态时，才能安全地销毁它。

    对于互斥体的加锁和解锁操作，我们一般使用以下三个函数：

    1. pthread_mutex_lock: 用于加锁互斥体。如果互斥体已经被其他线程锁定，调用线程将被阻塞，直到互斥体被解锁。

      函数签名：

      int pthread_mutex_lock(pthread_mutex_t* mutex);

    2. pthread_mutex_trylock: 尝试加锁互斥体，如果互斥体已经被其他线程锁定，则立即返回失败，不会阻塞当前线程。

      函数签名：

      int pthread_mutex_trylock(pthread_mutex_t* mutex);

    3. pthread_mutex_unlock: 用于解锁互斥体。解锁之后，其他线程可以再次加锁互斥体。

      函数签名：

      int pthread_mutex_unlock(pthread_mutex_t* mutex);

    参数mutex是我们需要加锁或解锁的互斥体对象，上述函数如果执行成功则返回 0，如果执行失败则返回一个错误码以表示具体的出错原因。具体的错误码随着互斥体对象属性类型的不同而不同。

    在设置互斥体对象的属性时，可以使用 pthread_mutexattr_t 类型的对象。下面是相关函数的签名：

    1. int pthread_mutexattr_init(pthread_mutexattr_t* attr);
      该函数用于初始化 pthread_mutexattr_t 对象，使其成为默认值。

    2. int pthread_mutexattr_destroy(pthread_mutexattr_t* attr);
      该函数用于销毁 pthread_mutexattr_t 对象，释放相关资源。

    在创建 pthread_mutex_t 对象之前，可以通过以下函数设置互斥体的属性：

    1. int pthread_mutexattr_settype(pthread_mutexattr_t* attr, int type);
      该函数用于设置互斥体的类型。参数 type 可以取以下值：

      • PTHREAD_MUTEX_NORMAL: 普通互斥体，默认值。
      • PTHREAD_MUTEX_ERRORCHECK: 错误检查互斥体，在同一线程中重复加锁会返回错误。
      • PTHREAD_MUTEX_RECURSIVE: 递归互斥体，允许同一线程多次加锁。

      注意：某些系统可能不支持所有类型，因此需要根据目标平台的支持情况来选择。

    互斥体对象的属性

    PTHREAD_MUTEX_NORMAL（普通锁）

    对于属性类型 PTHREAD_MUTEX_NORMAL，它表示普通锁，默认情况下互斥体的属性就是普通锁。当一个线程对普通锁加锁后，其他线程会被阻塞在 pthread_mutex_lock 调用处，直到持有锁的线程释放锁。

    以下是一个验证示例代码：

    #include <pthread.h>
#include <stdio.h>

pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;
int counter = 0;

void* increment(void* arg) {
    for (int i = 0; i < 100000; ++i) {
        pthread_mutex_lock(&mutex);
        counter++;
        pthread_mutex_unlock(&mutex);
    }
    return NULL;
}

int main() {
    pthread_t thread1, thread2;

    pthread_create(&thread1, NULL, increment, NULL);
    pthread_create(&thread2, NULL, increment, NULL);

    pthread_join(thread1, NULL);
    pthread_join(thread2, NULL);

    printf("Counter value: %d\n", counter);

    return 0;
}

    在上面的示例中，我们创建了两个线程 thread1 和 thread2，它们都执行 increment 函数，该函数对 counter 变量进行 100000 次加一操作。为了保证操作的原子性，我们使用互斥体 mutex 进行加锁和解锁操作。

    最终，我们打印出 counter 的值，预期结果应该是 200000。这是因为两个线程分别进行了 100000 次加一操作，互斥体保证了对 counter 的访问是线程安全的。

    但是对于一个已经加锁的普通锁（PTHREAD_MUTEX_NORMAL），如果同一个线程再次调用 pthread_mutex_lock 进行加锁操作，那么该线程会被阻塞在第二次的加锁操作处，形成死锁。

    以下是测试代码：

    #include <pthread.h>
#include <stdio.h>

pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

void* thread_func(void* arg) {
    printf("Thread starting...\n");

    // 第一次加锁
    pthread_mutex_lock(&mutex);
    printf("Thread acquired lock\n");

    // 第二次加锁，会导致死锁
    pthread_mutex_lock(&mutex);
    printf("Thread acquired lock again\n");

    pthread_mutex_unlock(&mutex);
    pthread_mutex_unlock(&mutex);

    printf("Thread exiting...\n");

    return NULL;
}

int main() {
    pthread_t thread;
    pthread_create(&thread, NULL, thread_func, NULL);

    pthread_join(thread, NULL);

    return 0;
}

    在上面的示例中，我们创建了一个线程 thread，该线程执行 thread_func 函数。在 thread_func 函数中，我们对互斥体 mutex 进行两次加锁操作。这样的操作会导致线程在第二次加锁操作处被阻塞，形成了死锁。
    在运行代码时，可以观察到程序会卡在第二次加锁操作处，并无法继续执行。这是因为同一个线程对已经加锁的普通锁再次进行加锁操作会导致线程自身无法解锁，从而形成了死锁。

    PTHREAD_MUTEX_ERRORCHECK（检错锁）

    对于属性类型 PTHREAD_MUTEX_ERRORCHECK，在同一个线程多次对已经加锁的互斥体对象进行加锁操作时，pthread_mutex_lock 函数会返回错误码 EDEADLK，表示检测到了死锁。

    #include <pthread.h>
#include <stdio.h>

pthread_mutexattr_t attr;
pthread_mutex_t mutex;

void* thread_func(void* arg) {
    printf("Thread starting...\n");

    // 第一次加锁
    pthread_mutex_lock(&mutex);
    printf("Thread acquired lock\n");

    // 第二次加锁，会返回 EDEADLK
    int result = pthread_mutex_lock(&mutex);
    if (result == EDEADLK) {
        printf("Thread detected deadlock\n");
    } else {
        printf("Thread acquired lock again\n");
        pthread_mutex_unlock(&mutex);
    }

    pthread_mutex_unlock(&mutex);

    printf("Thread exiting...\n");

    return NULL;
}

int main() {
    pthread_mutexattr_init(&attr);
    pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_ERRORCHECK);
    pthread_mutex_init(&mutex, &attr);

    pthread_t thread;
    pthread_create(&thread, NULL, thread_func, NULL);

    pthread_join(thread, NULL);

    pthread_mutex_destroy(&mutex);
    pthread_mutexattr_destroy(&attr);

    return 0;
}

    在上述代码中，我们首先使用 pthread_mutexattr_init 函数初始化互斥体属性对象 attr，然后使用 pthread_mutexattr_settype 函数设置属性类型为 PTHREAD_MUTEX_ERRORCHECK，接着使用 pthread_mutex_init 函数初始化互斥体 mutex，确保它使用了设置好的属性。

    在 thread_func 函数中，我们对互斥体进行两次加锁操作，并检查第二次加锁时返回的错误码。如果返回了 EDEADLK 错误码，则输出相应的提示信息。

    当运行代码时，线程在第二次加锁操作处返回了 EDEADLK 错误码，并输出了相应的提示信息，表示检测到了死锁。

    PTHREAD_MUTEX_RECURSIVE

    对于属性类型 PTHREAD_MUTEX_RECURSIVE，它允许同一个线程对持有的互斥体对象进行重复加锁操作，每次成功调用 pthread_mutex_lock 函数时，互斥体对象的锁引用计数会增加 1，而每次成功调用 pthread_mutex_unlock 函数时，锁引用计数会减少 1。只有当锁引用计数为 0 时，其他线程才能获得该锁；否则，其他线程调用 pthread_mutex_lock 时会阻塞在该函数调用处。
    这种属性类型的互斥体对象可以用于实现递归操作或者在嵌套函数调用中对共享资源进行保护。同一个线程在持有互斥体对象的同时可以多次调用加锁操作，只有在对应数量的解锁操作完成后，其他线程才能获得该锁。这种机制能够确保资源在嵌套函数调用中的正确访问和保护。
    需要注意的是，在使用 PTHREAD_MUTEX_RECURSIVE 属性的互斥体对象时，要确保加锁和解锁操作的次数匹配，否则可能导致死锁或其他错误。

    互斥体对象的使用要点的总结：

    1. 在实际开发中，同一个线程对一个互斥体对象反复加锁的场景非常少见。
    2. 为了避免在逻辑中忘记调用 pthread_mutex_unlock 导致死锁，或者在逻辑出口处出现大量重复的解锁代码，建议使用 RAII 技术（资源获取即初始化）来封装互斥体对象。这样可以确保在离开作用域时自动释放互斥体，避免忘记解锁的问题。

    RAII 技术可以通过使用 C++ 的智能指针或封装类来实现。在创建一个封装互斥体的类时，可以在构造函数中调用 pthread_mutex_lock，在析构函数中调用 pthread_mutex_unlock，这样可以确保互斥体对象在离开作用域时自动解锁。这种方式可以简化代码，提高代码的可维护性和可靠性。

    信号量

    信号量代表一定的资源数量，可以根据当前资源的数量按需唤醒指定数量的资源消费者线程，资源消费者线程一旦获取信号量，就会让资源减少指定的数量，如果资源数量减少为0，则消费者线程将全部处于挂起状态；当有新的资源到来时，消费者线程将继续被唤醒。另外，信号量有“资源有多份，可以同时被多个线程访问”的意思。
    在Linux中，信号量的常用API函数包括：

    1. int sem_init(sem_t *sem, int pshared, unsigned int value): 用于初始化一个命名或匿名信号量。参数sem是指向信号量对象的指针，pshared指定信号量的共享方式（0表示线程间共享，非0表示进程间共享），value是信号量的初始值。
    2. int sem_destroy(sem_t *sem): 用于销毁一个已初始化的信号量对象。参数sem是指向信号量对象的指针。
    3. int sem_wait(sem_t *sem): 对信号量进行P操作（等待），如果信号量的值大于0，则将其减1；如果值为0，则阻塞等待直到值大于0。
    4. int sem_trywait(sem_t *sem): 对信号量进行非阻塞的P操作（尝试等待），如果信号量的值大于0，则将其减1；如果值为0，则立即返回并返回错误码。
    5. int sem_timedwait(sem_t *sem, const struct timespec *abs_timeout): 对信号量进行定时等待的P操作，等待一定的时间，如果信号量的值大于0，则将其减1；如果值为0，则阻塞等待直到值大于0或超过指定的超时时间。
    6. int sem_post(sem_t *sem): 对信号量进行V操作（释放），将信号量的值加1，并唤醒等待该信号量的线程或进程。
    7. int sem_getvalue(sem_t *sem, int *sval): 获取信号量的当前值，并将其存储在sval指向的整型变量中。

    以上是Linux中信号量的常用API函数，它们定义在<semaphore.h>头文件中，用于实现进程间或线程间的同步和互斥操作。
    使用以上几个函数时还有几个需要注意的地方。

    1. 当多个线程调用 sem_wait 等函数等待同一个信号量时，只会有一个线程被唤醒，即只有一个线程可以成功获取到资源。这是因为这些函数在递减资源计数到1时会锁定信号量对象，确保只有一个线程可以通过。其他线程会在等待队列中等待，直到该线程释放信号量后才有机会获得资源。

    2. sem_wait、sem_trywait、sem_timedwait 函数在调用成功时返回值为0，表示成功获取到资源。而在调用失败时返回-1，并通过错误码 errno 来指示失败的原因。可以使用 errno 来判断具体的错误类型，例如 EAGAIN 表示资源当前不可用，ETIMEDOUT 表示超时等。

    3. sem_wait、sem_trywait、sem_timedwait 函数可以被Linux信号中断。当信号中断发生时，这些函数会立即返回，并将返回值设置为-1，同时错误码 errno 被设置为 EINTR。这种情况下，通常需要检查是否是由于信号中断而导致的返回，可以重新调用函数来处理信号中断的情况。

    4. 尽管这些函数的名称没有以 pthread_ 为前缀，但是实际使用时仍然需要链接 pthread 库。因为信号量是基于线程库实现的，因此需要链接相应的库以使用这些函数。

    下面是一个示例，演示了创建一个生产者线程和五个消费者线程，使用信号量来实现任务的生产和消费：

    #include <pthread.h>
#include <semaphore.h>
#include <iostream>
#include <queue>
#include <unistd.h>

// 全局变量
std::queue<int> taskQueue;  // 任务队列
pthread_mutex_t queueMutex; // 任务队列互斥锁
sem_t sem;                  // 信号量

// 生产者线程函数
void* producerThread(void* arg) {
    int taskId = 1;
    while (true) {
        // 生成任务
        int task = taskId++;

        // 将任务加入队列
        pthread_mutex_lock(&queueMutex);
        taskQueue.push(task);
        pthread_mutex_unlock(&queueMutex);

        // 增加信号量资源计数
        sem_post(&sem);

        sleep(1);
    }
    return NULL;
}

// 消费者线程函数
void* consumerThread(void* arg) {
    while (true) {
        // 等待信号量，等待可执行任务
        sem_wait(&sem);

        // 从队列中获取任务
        pthread_mutex_lock(&queueMutex);
        int task = taskQueue.front();
        taskQueue.pop();
        pthread_mutex_unlock(&queueMutex);

        // 处理任务
        std::cout << "Consumer thread " << pthread_self() << " processing task: " << task << std::endl;

        // 删除任务对象，避免内存泄漏
        // ...

        // 可以进行其他操作

        sleep(1);
    }
    return NULL;
}

int main() {
    // 初始化互斥锁和信号量
    pthread_mutex_init(&queueMutex, NULL);
    sem_init(&sem, 0, 0);

    // 创建生产者线程
    pthread_t producer;
    pthread_create(&producer, NULL, producerThread, NULL);

    // 创建消费者线程
    const int numConsumers = 5;
    pthread_t consumers[numConsumers];
    for (int i = 0; i < numConsumers; ++i) {
        pthread_create(&consumers[i], NULL, consumerThread, NULL);
    }

    // 等待生产者线程和消费者线程结束
    pthread_join(producer, NULL);
    for (int i = 0; i < numConsumers; ++i) {
        pthread_join(consumers[i], NULL);
    }

    // 销毁互斥锁和信号量
    pthread_mutex_destroy(&queueMutex);
    sem_destroy(&sem);

    return 0;
}

    在这个示例代码中，主函数创建了一个生产者线程和五个消费者线程。生产者线程循环生成任务，并将任务加入任务队列。每次生产任务后，通过 sem_post 函数增加信号量资源计数。

    消费者线程通过 sem_wait 函数等待可执行的任务。当信号量的资源计数大于0时，其中一个消费者线程会被唤醒，从任务队列中获取任务进行处理。在处理任务后，可以进行其他操作，如删除任务对象，避免内存泄漏。

    在调用sem_wait和sem_post时会对信号量对象进行加锁和解锁，为什么这里还需要使用一个互斥体呢？其实，这个互斥体是用来保护队列tasks的，因为多个线程会同时读写它。

    另外，为了确保线程的正确退出，主函数使用 pthread_join 函数等待生产者线程和消费者线程结束。

    Linux条件变量

    有人说 Linux 条件变量（Condition Variable）是最不容易用错的一种线程同步对象，确实是这样，但必须建立在对条件变量熟练应用的基础之上。

    为什么需要使用条件变量

    在实际应用中，我们常常会有类似如下的需求：

    // 以下是伪代码
// m 的类型是 pthread_mutex_t，已经初始化过了
int waitForTrue() {
    do {
        // 加锁互斥体，确保条件判断的原子性
        pthread_mutex_lock(&m);

        // 验证条件是否为 true
        // ...

        // 解锁互斥体，让其他线程有机会改变条件
        pthread_mutex_unlock(&m);

        // 睡眠 n 秒
        sleep(n);

    } while (condition is false);

    return 1;
}

    反复判断一个多线程的共享条件是否满足，一直到该条件满足为止。由于该条件被多个线程操作，因此在每次判断之前都需要进行加锁操作，判断完毕后需要进行解锁操作。但是，该逻辑存在严重的效率问题，假设解锁离开临界区后，其他线程修改了条件导致条件满足，则此时程序仍然需要睡眠n秒才能得到反馈。因此我们需要这样一种机制：某个线程 A 在条件不满足的情况下主动让出互斥体，让其他线程操作，线程 A在此处等待条件满足；一旦条件满足，线程 A就可以被立刻唤醒。线程 A之所以可以安心等待，依赖的是其他线程的协作，它确认会有一个线程在发现条件满足以后，向它发送信号并且让出互斥体。如果其他线程不配合（不发送信号、不让出互斥体），就无法实现上述效果。
    这就是为什么需要条件等待，但条件等待不是条件变量的全部功能。

    条件变量为什么要与互斥体对象结合使用

    为什么条件变量一定要与一个互斥体对象结合使用呢？我们假设条件变量不与互斥体对象结合，来看看是什么效果。以下是伪代码：

    // m的类型是 pthread_mutex_t，并且已经初始化过了，cv是条件变量

pthread_mutex_lock(&m);

while (condition_ is_false) {
    pthread_mutex_unlock(&m);

    // 解锁之后，等待之前，可能条件已经满足，信号已经发出，但是该信号可能被错过

    pthread_cond_wait(&cv, &m);

    pthread_mutex_lock(&m);
}

pthread_mutex_unlock(&m);

    在以上代码中，假设线程A在执行完pthread_mutex_unlock(&m)后CPU时间片被剥夺，此时另一个线程 B获得该互斥体对象 m，然后发送条件信号，等
    线程 A重新获得时间片后，由于该信号已经被错过，可能会导致线程A在代码第7行无限阻塞下去。造成这个问题的根源是释放互斥体对象与条件变量等待唤醒不是原子操作，即解锁和等待这两个步骤必须在同一个原子操作中，才能确保cond_wait在唤醒之前不会有其他线程获得这个互斥体对象。

    条件变量的使用

    对条件变量进行初始化和销毁，可以使用以下 API 函数：

    1. pthread_cond_init：用于初始化条件变量。

      • 函数原型：int pthread_cond_init(pthread_cond_t *cond, const pthread_condattr_t *attr);
      • 参数 cond 是指向条件变量对象的指针。
      • 参数 attr 是指向条件变量属性对象的指针，可以为 NULL 表示使用默认属性。
      • 返回值：成功返回 0，失败返回错误码。

    2. pthread_cond_destroy：用于销毁条件变量。

      • 函数原型：int pthread_cond_destroy(pthread_cond_t *cond);
      • 参数 cond 是指向条件变量对象的指针。
      • 返回值：成功返回 0，失败返回错误码。

    在 Linux 系统中，pthread_cond_t 表示条件变量的数据类型。可以使用如下方式初始化一个条件变量：

    #include <pthread.h>

pthread_cond_t cv = PTHREAD_COND_INITIALIZER;

    上述代码将条件变量 cv 初始化为静态分配的条件变量对象，并使用宏 PTHREAD_COND_INITIALIZER 进行初始化。这种方式适用于静态分配
    使用示例：

    #include <pthread.h>

pthread_cond_t cv; // 声明条件变量对象

int main() {
    // 初始化条件变量
    int result = pthread_cond_init(&cv, NULL);
    if (result != 0) {
        // 处理初始化失败的情况
    }

    // 使用条件变量

    // 销毁条件变量
    result = pthread_cond_destroy(&cv);
    if (result != 0) {
        // 处理销毁失败的情况
    }

    return 0;
}

    注意：在使用条件变量之前，需要先初始化它，使用完毕后再进行销毁操作。确保在销毁条件变量之前，没有线程在等待或使用该条件变量。

    可以使用以下 API 函数等待条件变量被唤醒：

    #include <pthread.h>

int pthread_cond_wait(pthread_cond_t *cond, pthread_mutex_t *mutex);

    函数 pthread_cond_wait 用于等待条件变量的信号，并在等待期间自动释放关联的互斥体锁，并在被唤醒后重新获得锁。函数的参数 cond 是指向条件变量的指针，mutex 是指向互斥体对象的指针。

    调用 pthread_cond_wait 时，会进行以下操作：

    1. 首先，函数会自动释放 mutex 指向的互斥体锁，以允许其他线程访问临界区。
    2. 然后，线程将进入等待状态，直到接收到条件变量的信号。
    3. 当条件变量的信号被发送并接收到时，线程将重新获得 mutex 指向的互斥体锁，并从 pthread_cond_wait 函数返回。

    需要注意的是，pthread_cond_wait 函数需要在持有互斥体锁的情况下调用，否则行为是未定义的。在调用 pthread_cond_wait 前后，通常需要使用互斥体来保护条件的判断和修改，以确保线程安全性。

    使用示例：

    #include <pthread.h>

pthread_cond_t cv;
pthread_mutex_t mutex;

void* thread_function(void* arg) {
    // 在获取锁之前做一些准备工作

    pthread_mutex_lock(&mutex);

    // 检查条件是否满足
    while (condition_is_false) {
        // 等待条件变量的信号，自动释放互斥体锁
        pthread_cond_wait(&cv, &mutex);

        // 在被唤醒后重新获得互斥体锁，继续检查条件
    }

    // 条件满足后的处理

    pthread_mutex_unlock(&mutex);

    return NULL;
}

    在上述示例中，线程在等待条件变量的信号时会自动释放互斥体锁，并在被唤醒后重新获得锁，以继续执行后续操作。

    以下是可以唤醒等待在条件变量上的线程的 API 函数：

    #include <pthread.h>

int pthread_cond_signal(pthread_cond_t *cond);
int pthread_cond_broadcast(pthread_cond_t *cond);
    • pthread_cond_signal 函数用于唤醒等待在条件变量上的一个线程。如果有多个线程在等待条件变量上，只会唤醒其中一个线程，具体唤醒哪个线程是不确定的。
    • pthread_cond_broadcast 函数用于唤醒等待在条件变量上的所有线程。

    这两个函数的参数 cond 是指向条件变量的指针。pthread_cond_signal和pthread_cond_broadcast均返回0，反之均返回具体的错误码值。

    调用 pthread_cond_signal 或 pthread_cond_broadcast 函数后，等待在条件变量上的线程将被唤醒，并重新开始竞争互斥体锁。被唤醒的线程将继续执行等待后的代码，直到重新获得互斥体锁。

    需要注意的是，为了确保等待线程能够正确地接收到信号，通常在发送信号之前需要先获得相关的互斥体锁，以保证操作的原子性。

    使用示例：

    #include <pthread.h>

pthread_cond_t cv;
pthread_mutex_t mutex;

void signal_condition() {
    pthread_mutex_lock(&mutex);

    // 发送条件变量的信号，唤醒等待的线程
    pthread_cond_signal(&cv);

    pthread_mutex_unlock(&mutex);
}

void broadcast_condition() {
    pthread_mutex_lock(&mutex);

    // 发送条件变量的广播信号，唤醒所有等待的线程
    pthread_cond_broadcast(&cv);

    pthread_mutex_unlock(&mutex);
}

    在上述示例中，通过调用 pthread_cond_signal 函数或 pthread_cond_broadcast 函数，可以唤醒等待在条件变量上的线程。这些线程会在被唤醒后竞争互斥体锁，并继续执行等待后的代码。

    在使用 pthread_cond_wait 函数时，需要注意以下两点：

    1. 调用 pthread_cond_wait 前需要对互斥体进行加锁操作：在等待条件时，线程需要持有与条件变量关联的互斥体锁。因此，在调用 pthread_cond_wait 之前，通常需要使用 pthread_mutex_lock 函数对互斥体进行加锁操作，确保在等待条件时互斥体是被当前线程所持有。这样可以避免在等待过程中出现竞态条件。

    2. 收到条件信号时，pthread_cond_wait 函数会返回并对其绑定的互斥体进行加锁操作：当等待的条件满足并且收到条件信号时，pthread_cond_wait 函数会返回，并且在返回时会对其绑定的互斥体进行加锁操作。这是为了确保线程在返回后可以安全地访问共享资源。因此，在 pthread_cond_wait 的下面，通常会有对互斥体进行解锁的操作，以允许其他线程获得互斥体锁，并且可以继续执行需要保护的临界区代码。

    条件变量的虚假唤醒

    在多线程编程中，存在一种称为 “虚假唤醒”（spurious wakeup）的情况。即使没有收到条件信号，线程有时也可能被唤醒。因此，在使用条件变量时，需要使用 while 循环来再次检查条件是否满足，以防止虚假唤醒导致的错误行为。
    虚假唤醒这看起来像个Bug，但它在Linux系统中是实实在在存在的。假设pthread_cond_wait函数被信号中断，则在 pthread_cond_wait 函数返回之后，到重新调用之前，pthread_cond_signal 或pthread_cond_broadcast函数可能已被调用。一旦错失该信号，则可能由于条件信号不再产生，再次调用 thread_cond_wait 函数将导致程序无限等待。为了避免这种情况发生，我们宁可虚假唤醒，也不能再次调用pthread_cond_wait函数，以免陷入无穷等待中。
    这种虚假唤醒可能发生在以下情况下：

    1. 平台实现：不同的操作系统或编译器对条件变量的实现可能会导致虚假唤醒的问题。
    2. 信号中断：当线程被信号中断唤醒时，可能会发生虚假唤醒。信号中断会打断线程的阻塞状态，使其从 pthread_cond_wait 返回，但并没有实际的条件变化发生。
    3. 在条件满足时发送信号，但等到调用pthread_cond_wait函数的线程得到CPU时间片时，条件又再次不满足了。
      通过使用 while 循环再次检查条件，可以确保只有在条件满足时线程才继续执行，否则会再次进入等待状态。这样可以防止因为虚假唤醒而引发的错误行为。
      除了上面的信号因素，还存在一些情况：在条件满足时发送信号，但等到调用pthread_cond_wait函数的线程得到CPU时间片时，条
      件又再次不满足了。
      以下是示例代码:
    pthread_mutex_t mutex;
pthread_cond_t cv;
int condition = 0;

void* thread_function(void* arg) {
    pthread_mutex_lock(&mutex);
    
    // 使用 while 循环来再次检查条件
    while (condition == 0) {
        pthread_cond_wait(&cv, &mutex);
    }
    
    // 条件满足后的处理
    // ...

    pthread_mutex_unlock(&mutex);
    return NULL;
}

void signal_condition() {
    pthread_mutex_lock(&mutex);
    
    // 更新条件
    condition = 1;
    
    // 发送条件变量的信号，唤醒等待的线程
    pthread_cond_signal(&cv);
    
    pthread_mutex_unlock(&mutex);
}

    通过在 while 循环中再次检查条件，即使出现虚假唤醒的情况，线程也会再次等待条件满足，以确保正确的行为。

    条件变量信号丢失的问题

    如果一个条件变量信号在产生时（调用pthread_cond_signal或pthread_cond_broadcast），没有相关线程调用 pthread_cond_wait 捕获该信号，该信号就会永久丢失，再次调用pthread_cond_wait 会导致永久阻塞。我们在设计条件变量信号只会产生一次的逻辑中尤其需要注意这种情况。举个例子，假设现在某个程序中有一批等待条件变量的线程，和一个只产生一次条件变量信号的线程。为了让等待条件变量的线程能够正常运行而不阻塞，编写这段逻辑时一定要确保等待的线程在产生条件变量信号的线程发送条件信号之前调用pthread_cond_wait。

    Linux读写锁

    读写锁的应用场景

    在实际应用中，对共享变量的访问大多有个特点：在大多数情况下，线程只是读取共享变量的值，只在极少数情况下才会真正修改共享变量的值。对于这种情况，读请求之间无须同步，它们之间的并发访问是安全的。然而写请求必须锁住读请求和其他写请求。这在实际应用中是存在的，例如读取一个全局对象的状态属性，这个状态属性的值一般不会变化，偶尔才会被修改。如果使用互斥体完全阻止读请求并发，则会造成性能的损失。

    读写锁的相关API

    在Linux系统中，可以使用以下系统API函数来初始化和销毁读写锁：

    1. 初始化读写锁：int pthread_rwlock_init(pthread_rwlock_t* rwlock, const pthread_rwlockattr_t* attr)

      • 参数 rwlock：指向要初始化的读写锁对象的指针。
      • 参数 attr：指向读写锁属性的指针，通常设置为NULL表示使用默认属性。
      • 返回值：若成功初始化读写锁，则返回0；否则返回错误码。

    2. 销毁读写锁：int pthread_rwlock_destroy(pthread_rwlock_t* rwlock)

      • 参数 rwlock：指向要销毁的读写锁对象的指针。
      • 返回值：若成功销毁读写锁，则返回0；否则返回错误码。

    当然，如果不需要动态创建或者设置非默认属性的读写锁对象，则也可以使用如下语法初始化一个读写锁对象：

    pthread_rwlock_t rwlock = PTHREAD_RWLOCK_INITIALIZER;

    以下是请求读锁和写锁的系统API接口：

    请求读锁的系统API接口：

    1. int pthread_rwlock_rdlock(pthread_rwlock_t *rwlock)：请求读取锁。如果读取锁可用，则将读取锁分配给调用线程，否则线程将阻塞，直到读取锁可用。
    2. int pthread_rwlock_tryrdlock(pthread_rwlock_t *rwlock)：尝试请求读取锁。如果读取锁可用，则将读取锁分配给调用线程并返回0，否则立即返回EBUSY，不会阻塞线程。
    3. int pthread_rwlock_timedrdlock(pthread_rwlock_t *rwlock, const struct timespec *abs_timeout)：在指定的超时时间内请求读取锁。如果在超时时间内读取锁可用，则将读取锁分配给调用线程并返回0，否则返回ETIMEDOUT。

    请求写锁的系统API接口：

    1. int pthread_rwlock_wrlock(pthread_rwlock_t *rwlock)：请求写入锁。如果写入锁可用，则将写入锁分配给调用线程，否则线程将阻塞，直到写入锁可用。
    2. int pthread_rwlock_trywrlock(pthread_rwlock_t *rwlock)：尝试请求写入锁。如果写入锁可用，则将写入锁分配给调用线程并返回0，否则立即返回EBUSY，不会阻塞线程。
    3. int pthread_rwlock_timedwrlock(pthread_rwlock_t *rwlock, const struct timespec *abs_timeout)：在指定的超时时间内请求写入锁。如果在超时时间内写入锁可用，则将写入锁分配给调用线程并返回0，否则返回ETIMEDOUT。

    这些API函数可以用于在多线程环境中对读写锁进行加锁操作。

    读写锁的使用规则如下：

    读模式（共享模式）：

    • 如果当前读写锁已经被某线程以读模式占有，其他线程可以立即获取读锁，而不会被阻塞。
    • 如果当前读写锁已经被某线程以写模式占有，其他线程请求读锁会被阻塞，直到写锁被释放。

    写模式（独占模式）：

    • 如果当前读写锁已经被某线程以写模式占有，其他线程无论是请求读锁还是写锁，都会被阻塞，直到写锁被释放。
    • 在写模式下，不允许其他线程获取读锁或写锁。

    通过读写锁，可以实现对共享资源的并发读取和独占写入的控制，提高系统的并发性和吞吐量。

    释放读锁和写锁都使用同一个接口

    pthread_rwlock_unlock

    该函数用于释放当前线程持有的读锁或写锁。
    无论是请求读锁还是写锁，都提供了 trylock 的功能（pthread_rwlock_tryrdlock 和pthread_rwlock_trywrlock），调用线程不会阻塞，而是立即返回。如果能成功获得读锁或者写锁，则函数返回0，如果不能获得读锁或写锁，则函数返回非0值，此时错误码errno是EBUSY。当然，无论是请求读锁还是写锁，都提供了限时等待功能。如果不能获取读写锁，则会陷入阻塞，最多等待到参数 abstime设置的时间，如果此时仍然无法获得锁，则返回，错误码errno是ETIMEOUT。

    读写锁的属性

    pthread_rwlock_init函数我们可以知道第2个参数可以设置读写锁的属性，读写锁的属性类型是pthread_rwlockattr_t。glibc引入了如下接口来查询和改变读写锁的类型：

    pthread_rwlockattr_init: 初始化读写锁属性对象。
pthread_rwlockattr_destroy: 销毁读写锁属性对象。
pthread_rwlockattr_getkind_np: 获取读写锁的类型。
pthread_rwlockattr_setkind_np: 设置读写锁的类型。

    其中，函数名中的 _np 表示这些函数是非标准的扩展函数，属于 GNU C 库 (glibc) 的特定扩展。这些函数允许查询和设置读写锁的类型属性，以控制读写锁的行为。

    pthread_rwlockattr_setkind_np 函数的第二个参数 pref 用于设置读写锁的类型，它可以取以下值：

    • PTHREAD_RWLOCK_PREFER_READER_NP: 优先读者模式。在有读者等待时，尽量让其他读者获得读锁，以提高读取操作的并发性。
    • PTHREAD_RWLOCK_PREFER_WRITER_NP: 优先写者模式。在有写者等待时，尽量让写者获得写锁，以避免读者长时间等待。
    • PTHREAD_RWLOCK_PREFER_WRITER_NONRECURSIVE_NP: 优先非递归写者模式。在有写者等待时，只允许以非递归方式获取写锁，以避免写者之间发生死锁情况。

    通过设置读写锁的类型，可以根据应用场景选择适当的读写锁行为，以提高并发性或避免死锁问题。需要注意的是，这些取值是非标准的扩展，具体实现可能因操作系统和库版本而有所差异。在使用时，应注意兼容性和可移植性。

    读写锁应用示例

    读写锁应用示例如下：

    #include <pthread.h>
#include <unistd.h>
#include <iostream>

int resourceID = 0;
pthread_rwlock_t myrwlock;

void* read_thread(void* param)
{	
    while (true)
    {
        //请求读锁
        pthread_rwlock_rdlock(&myrwlock);

        std::cout << "read thread ID: " << pthread_self() << ", resourceID: " << resourceID << std::endl;
                
        //使用睡眠模拟读线程读的过程消耗了很久的时间
        sleep(1);
                
        pthread_rwlock_unlock(&myrwlock);
    }
    
    return NULL;
}

void* write_thread(void* param)
{
    while (true)
    {
        //请求写锁
        pthread_rwlock_wrlock(&myrwlock);

        ++resourceID;
        std::cout << "write thread ID: " << pthread_self() << ", resourceID: " << resourceID << std::endl;
                
        //使用睡眠模拟读线程读的过程消耗了很久的时间
        sleep(1);
                
        pthread_rwlock_unlock(&myrwlock);
    }
    
    return NULL;
}

int main()
{
    pthread_rwlock_init(&myrwlock, NULL);

    //创建5个请求读锁线程
    pthread_t readThreadID[5];
    for (int i = 0; i < 5; ++i)
    {
        pthread_create(&readThreadID[i], NULL, read_thread, NULL);
    }
    
    //创建一个请求写锁线程
    pthread_t writeThreadID;
    pthread_create(&writeThreadID, NULL, write_thread, NULL);

    pthread_join(writeThreadID, NULL);
    
    for (int i = 0; i < 5; ++i)
    {
        pthread_join(readThreadID[i], NULL);
    }
    
    pthread_rwlock_destroy(&myrwlock);

    return 0;
}
