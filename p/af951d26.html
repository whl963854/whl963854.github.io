    线程局部存储

    对于1个存在多个线程的进程来说，有时需要每个线程都自己操作自己的这份数据。这有点类似于 C++类的实例属性，每个实例对象操作的都是自己的属性。我们把这样的数据称为线程局部存储（ThreadLocal Storage，TLS），将对应的存储区域称为线程局部存储区。

    Windows的线程局部存储

    在 Windows 操作系统中，TLS（Thread Local Storage，线程局部存储）被划分为一定数量的块，每个块通过一个索引值来进行访问。这个数量由宏定义 TLS_MINIMUM_AVAILABLE 来表示，默认值为 64。

    在 Windows 的头文件 winnt.h 中，可以找到以下定义：

    #define TLS_MINIMUM_AVAILABLE 64

    这意味着系统至少提供了 64 个 TLS 块供开发者使用。开发者可以使用这些块来存储线程本地的数据，每个线程都有自己独立的一份数据副本。通过 TLS，可以在多线程环境中实现线程安全的数据访问，每个线程可以独立地操作自己的数据，而不会互相干扰。
    需要注意的是，虽然最少提供了 64 个 TLS 块，但实际上可以使用更多的 TLS 块。开发者可以通过函数 TlsAlloc 来分配一个新的 TLS 索引值，用于访问额外的 TLS 块。每个线程都可以通过索引值来访问自己对应的 TLS 块。
    总结起来，TLS 在 Windows 中提供了一种机制，用于实现线程本地数据的存储和访问。通过 TLS，可以方便地在多线程环境中管理线程私有的数据。

    示意图如下：

    在 Windows 中，可以使用 TlsAlloc 函数来获取一个可用的线程局部存储（TLS）块的索引。以下是使用 TlsAlloc 函数的示例代码：

    #include <Windows.h>

DWORD dwTlsIndex;  // 全局变量，用于存储 TLS 块的索引

void InitializeTls()
{
    // 获取一个可用的 TLS 块索引
    dwTlsIndex = TlsAlloc();
    if (dwTlsIndex == TLS_OUT_OF_INDEXES)
    {
        // 获取索引失败，处理错误
        // ...
    }
}

void CleanupTls()
{
    // 释放先前分配的 TLS 块索引
    TlsFree(dwTlsIndex);
}

void SetTlsData(LPVOID lpData)
{
    // 将指定的数据存储到当前线程的 TLS 块中
    TlsSetValue(dwTlsIndex, lpData);
}

LPVOID GetTlsData()
{
    // 从当前线程的 TLS 块中获取数据
    return TlsGetValue(dwTlsIndex);
}
    • 在上述示例中，InitializeTls 函数用于初始化 TLS，调用 TlsAlloc 函数获取一个可用的 TLS 块索引，并将其保存在全局变量 dwTlsIndex 中。
    • CleanupTls 函数用于在程序结束或不再需要 TLS 时释放先前分配的 TLS 块索引，调用 TlsFree 函数进行释放。
    • SetTlsData 函数用于将指定的数据存储到当前线程的 TLS 块中，调用 TlsSetValue 函数进行设置。
    • GetTlsData 函数用于从当前线程的 TLS 块中获取数据，调用 TlsGetValue 函数进行获取。
      需要注意的是，TLS 块中存储的是 LPVOID 类型的指针，可以根据需要将不同类型的数据进行类型转换。另外，每个线程都有独立的一份 TLS 数据，相互之间不会共享。

    一旦 TlsAlloc 函数调用成功并返回一个有效的索引，就可以使用以下两个 API 函数在该索引指向的 TLS 内存块中存储和获取数据：

    1. TlsSetValue 函数：用于将数据存储到指定索引的 TLS 内存块中。

      BOOL TlsSetValue(DWORD dwTlsIndex, LPVOID lpTlsValue);
      • dwTlsIndex：要存储数据的 TLS 索引。
      • lpTlsValue：要存储的数据指针。

    2. TlsGetValue 函数：用于从指定索引的 TLS 内存块中获取数据。

      LPVOID TlsGetValue(DWORD dwTlsIndex);
      • dwTlsIndex：要获取数据的 TLS 索引。

    当不再需要使用一个 TLS 索引及其对应的内存块时，可以使用 TlsFree 函数来释放它们：

    BOOL TlsFree(DWORD dwTlsIndex);
    • dwTlsIndex：要释放的 TLS 索引。

    除了使用 Win32 API 函数来管理线程局部存储外，Microsoft VC++ 编译器也提供了一种更方便的方法来定义线程局部变量，即使用 __declspec(thread) 关键字。

    具体用法如下所示：

    __declspec(thread) <variable-type> <variable-name>;

    其中，<variable-type> 是要定义的线程局部变量的类型，<variable-name> 是变量的名称。

    来看一个具体的例子：

    #include <windows.h>
#include <iostream>

__declspec(thread) int g_mydata = 1;

DWORD WINAPI workerThreadProc1(LPVOID lpThreadParameter)
{
    while (true)
    {
        ++g_mydata;
        Sleep(1000);
    }
    return 0;
}

DWORD WINAPI workerThreadProc2(LPVOID lpThreadParameter)
{
    while (true)
    {
        std::cout << "g_mydata = " << g_mydata << ", ThreadID = " << GetCurrentThreadId() << std::endl;
        Sleep(1000);
    }
    return 0;
}

int main()
{
    HANDLE hWorkerThreads[2];
    hWorkerThreads[0] = CreateThread(NULL, 0, workerThreadProc1, NULL, 0, NULL);
    hWorkerThreads[1] = CreateThread(NULL, 0, workerThreadProc2, NULL, 0, NULL);

    CloseHandle(hWorkerThreads[0]);
    CloseHandle(hWorkerThreads[1]);

    while (true)
    {
        Sleep(1000);
    }
    return 0;
}

    在以上代码中，全局变量 g_mydata是一个线程局部变量，因此在该进程中每一个线程都会拥有这样一个变量副本。由于是不同的副本，所以在WorkerThreadProc1中，这个变量不断递增，对WorkerThreadProc2的g_mydata不会造成任何影响，因此其值始终是1。

    Linux的线程局部存储

    在Linux上，NTPL（Native POSIX Thread Library）提供了一套函数接口来实现线程局部存储（Thread-Local Storage，TLS）功能。这些函数接口包括：

    1. int pthread_key_create(pthread_key_t* key, void (*destructor)(void*))

      • 参数key：指向线程局部存储键的指针，用于存储创建的键。
      • 参数destructor：线程局部存储键关联值的清理函数指针。

      返回值：若成功，返回0；否则，返回错误码。

    2. int pthread_key_delete(pthread_key_t key)

      • 参数key：要删除的线程局部存储键。

      返回值：若成功，返回0；否则，返回错误码。

    3. int pthread_setspecific(pthread_key_t key, const void* value)

      • 参数key：要设置值的线程局部存储键。
      • 参数value：要与键关联的值。

      返回值：若成功，返回0；否则，返回错误码。

    4. void* pthread_getspecific(pthread_key_t key)

      • 参数key：要获取值的线程局部存储键。

      返回值：与键关联的值。

    5. int pthread_once(pthread_once_t* once_control, void (*init_routine)(void))

      • 参数once_control：用于标记一次性初始化是否已完成的控制变量。
      • 参数init_routine：一次性初始化的回调函数指针。

      返回值：若成功，返回0；否则，返回错误码。

    需要注意的是，函数签名中的数据类型可能会根据具体的编译环境和操作系统有所不同，上述签名是一般情况下的表示。请根据实际情况参考系统的文档和头文件定义。

    通过使用这些函数接口，可以在Linux上实现线程局部存储，每个线程可以独立地存储和访问其自己的线程局部数据。这对于需要在线程之间保留独立数据的情况非常有用。
    参数destructor是一个自定义函数指针，用于在线程退出时对线程局部存储的值进行清理操作。其函数签名如下：

    void destructor(void* value);

    函数指针destructor指向的函数接受一个void*类型的参数，该参数表示线程局部存储的值。在线程退出时，系统会调用注册的清理函数来处理线程局部存储的值。清理函数可以根据需要进行适当的处理，例如释放内存、关闭文件等。
    线程终止时，如果key关联的值不是NULL，那么NTPL会自动执行定义的destructor函数；如果无须解构，则可以将destructor设置为NULL。

    下面看一个例子：

    #include <pthread.h>
#include <stdio.h>
#include <stdlib.h>

pthread_key_t thread_log_key;

void write_to_thread_log(const char* message) {
    if (message == NULL)
        return;
    FILE* logfile = (FILE*)pthread_getspecific(thread_log_key);
    fprintf(logfile, "%s\n", message);
    fflush(logfile);
}

void close_thread_log(void* logfile) {
    char logfilename[128];
    sprintf(logfilename, "close logfile: thread%ld.log\n", (unsigned long)pthread_self());
    printf(logfilename);
    fclose((FILE*)logfile);
}

void* thread_function(void* args) {
    char logfilename[128];
    sprintf(logfilename, "thread%ld.log", (unsigned long)pthread_self());
    FILE* logfile = fopen(logfilename, "w");
    if (logfile != NULL) {
        pthread_setspecific(thread_log_key, logfile);
        write_to_thread_log("Thread starting...");
    }
    return NULL;
}

int main() {
    pthread_t threadIDs[5];
    pthread_key_create(&thread_log_key, close_thread_log);
    for (int i = 0; i < 5; ++i)
        pthread_create(&threadIDs[i], NULL, thread_function, NULL);
    for (int i = 0; i < 5; ++i)
        pthread_join(threadIDs[i], NULL);
    return 0;
}

    上述程序一共创建了5个线程，每个线程都会自己生成一个日志文件，每个线程都将自己的日志写入自己的文件中。在线程执行结束时，会关闭打开的日志文件句柄。程序运行结果如下所示：

    和Windows一样，Linux gcc编译器也提供了一个关键字__thread用于定义线程局部变量。例如：

    __thread int val=0

    再看一个例子：

    #include <stdio.h>

__thread int thread_data;

void* thread_function(void* args) {
    thread_data = 42;
    printf("Thread data: %d\n", thread_data);
    return NULL;
}

int main() {
    pthread_t thread_id;
    pthread_create(&thread_id, NULL, thread_function, NULL);
    pthread_join(thread_id, NULL);
    
    // 在主线程中访问线程局部变量
    printf("Main thread data: %d\n", thread_data);
    
    return 0;
}

    在上述示例中，每个线程都有自己的thread_data变量副本，可以独立地对其进行操作。在主线程中也可以访问线程局部变量。
    需要注意的是，使用__thread定义的线程局部变量仅适用于线程函数内部和主线程中，不能在多个线程之间共享。

    C++11的thread_local关键字

    C++11标准引入了关键字thread_local，用于定义线程局部变量。使用thread_local关键字可以在C++代码中声明一个变量，使其具有线程局部存储的特性。它是一个和 static、extern 同级的变量存储说明，有 thread_local 标记的变量在每个线程里都会有一个独立的副本，是“线程独占”的，所以就不会有竞争读写的问题。有了这个关键字，使用线程局部存储的代码就可以同时在Windows 和 Linux 上运行了。示例如下：

    #include <iostream>

thread_local int thread_data;

void thread_function() {
    thread_data = 42;
    std::cout << "Thread data: " << thread_data << std::endl;
}

int main() {
    std::thread t1(thread_function);
    std::thread t2(thread_function);

    t1.join();
    t2.join();
    
    // 在主线程中访问线程局部变量
    std::cout << "Main thread data: " << thread_data << std::endl;

    return 0;
}

    在上述示例中，thread_data是一个线程局部变量，每个线程都有自己的变量副本。在各个线程函数中，可以独立地对thread_data进行操作。在主线程中也可以访问线程局部变量。

    强调

    最后，关于线程局部存储变量，需要再强调两点。

    • 对于线程变量，每个线程都会有该变量的一个拷贝，互不影响，该局部变量一直存在，直到线程退出。
    • 系统的线程局部存储区域的内存空间并不大，所以尽量不要用这个空间存储大的数据块，如果不得不使用大的数据块，则可以将大的数据块存储在堆内存中，再将该堆内存的地址指针存储在线程局部存储区域。
